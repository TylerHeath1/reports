Change Intelligence

Three product concepts for the OS Hub Plus tier · User Story 1 · Prototype v0.1
User Story 1

"Tell me when my supplier data changes in a way that matters."

Brands pay for OS Hub data to monitor their supplier networks continuously — not just at onboarding. When a meaningful change happens (a certification is revoked, an ownership transfer lands a facility under an unvetted entity, an NGO reports an unregistered production site), there is no reliable mechanism to surface it to the right person at the right moment. Success metric: a user opens this product and identifies the one thing they need to act on today in under 90 seconds. Biggest risk: alert fatigue — this product fails if users learn to ignore it.

Primary persona: compliance / responsible sourcing manager Price: $15K–$30K / year Target ARR: $500K across 20–35 customers Pilots: Amazon · Disney · ASDA · Worldly

We're not building three products. We're building one.

Architecture insight

One foundation. Three surfaces. Two delivery channels.

The change feed is the primitive. The alert dashboard is the feed with rules + personalization layered on top. The supplier table is the feed grouped by supplier with a before/after join. The API and CSV export are the same events in different packaging. We don't build three products — we build one event log and five thin adapters.

Layer 1 · Foundation

Change event log

Every edit becomes a structured event: {os_id, field, before, after, contributor, timestamp}.

Layer 2 · Context

Rules + personalization

Severity rules tag events High/Med/Low. Personalization scopes them to starred suppliers, Tier 1 lists, recent searches.

Layer 3 · Surfaces

Feed · Alert · Table · API · CSV

Five thin adapters over the same stream. Each customer can use one, two, or all five depending on workflow.

MVP sequencing

1

Change feed (Concept 2) — build first

The only concept that can ship on current data infrastructure without new engineering. The system already tracks that something changed; a feed only needs to render events in order with severity. It buys time to collect real user behavior before committing to a triage model. It also hedges the field-level detection kill criterion — if engineering confirms more than one quarter of build, we can still ship the feed and sell against it.

2

Alert-first (Concept 1) — build second

Once behavioral data from the feed tells us which change types correlate with user action, the triage model has a foundation. Shipping it cold means guessing, and the cost of being wrong is the product's core failure mode (alert fatigue). Concept 1 is Concept 2 after twelve months of telemetry and field-level detection live.

3

Supplier table + diff (Concept 3) — ship as a power-user view inside the above

Valuable to auditors, but auditors are not the economic buyer — VP Sustainability writes the cheque. Package the diff expansion and audit trail as the "evidence export" feature of Concepts 1 or 2 and use it to unlock the Worldly / service-provider reseller channel.

Risks to design against

High severity · High likelihood

Field-level change detection not yet built

The system tracks that something changed, not what or why. Engineering build required; not on current roadmap. Confirm feasibility before advancing.

High severity · Medium likelihood

Pricing resistance from existing customers

Customers may expect change signals as part of their current subscription. Requires clear tier differentiation before launch.

Medium severity · Medium likelihood

Contributor update frequency is inconsistent

Change alerts only work if contributors update regularly. MSI mandates help but don't guarantee frequency across all sectors.
